## Q1 — Does `public/_headers` apply to SSR (Worker-generated) responses, and is `_headers`/`_redirects` even supported for Workers Static Assets?

**Source (Tier 1):** https://developers.cloudflare.com/workers/static-assets/headers/ (last updated Apr 23, 2026)
Companion: https://developers.cloudflare.com/workers/static-assets/redirects/

### Answer
- **`_headers` and `_redirects` ARE supported for Workers Static Assets** (not just Cloudflare Pages). The `_headers` file goes in your static asset directory (your `public/`); Wrangler parses it and applies the rules to **static asset responses**.
- **They apply ONLY to static-asset responses. They are NOT applied to responses generated by your Worker code** — including SSR HTML. This is stated twice on the page, explicitly calling out SSR:

> "Custom headers defined in the `_headers` file **are not applied to responses generated by your Worker code**, even if the request URL matches a rule defined in `_headers`. If you use a server-side rendered (SSR) framework, have configured `assets.run_worker_first`, or otherwise use a Worker script, you will likely need to attach any custom headers you wish to apply directly within that Worker script."

> "If you are server-side rendering (SSR) or using a Worker to generate responses in any other way and wish to attach security headers, the headers should be sent from the Worker's `Response` instead of using a `_headers` file."

### What this means for Granlund Grove (concrete)
Your SSR HTML document responses are produced by `src/server.ts` -> `@tanstack/react-start/server-entry`. Therefore:
- `Content-Security-Policy`, `Strict-Transport-Security`, `Permissions-Policy` from `public/_headers` **will be absent on the SSR HTML.**
- They WILL appear on true static assets served by the asset system (fonts under `/fonts/*`, `/favicon.svg`, `/og-cover.jpg`, `/resume/*.pdf`, `robots.txt`).
- The headers your Worker already sets in `applyHeaders()` (`X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`) DO appear on SSR HTML — those are fine. The gap is CSP / HSTS / Permissions-Policy.

**Verification you can run after deploy:** `curl -sI https://<domain>/ | grep -i 'content-security-policy\|strict-transport'` — today this returns nothing for the HTML document.

**Recommendation:** Set CSP, HSTS, Permissions-Policy in `applyHeaders()`. You can keep them in `_headers` too (belt-and-suspenders for static assets), but the Worker is the source of truth for the document. To avoid drift, consider a single shared constant imported by both build step and Worker.

## Q2 — With `nodejs_compat` and compat date 2025-09-24, will `process.env.RESEND_API_KEY` resolve, or only `env.RESEND_API_KEY`?

**Source (Tier 1):** https://developers.cloudflare.com/workers/configuration/environment-variables/ (last updated Apr 23, 2026), section "Environment variables and Node.js compatibility".

### Answer: YES, `process.env.RESEND_API_KEY` will resolve at runtime.

Verbatim:
> "When you enable `nodejs_compat` and the `nodejs_compat_populate_process_env` compatibility flag (**enabled by default for compatibility dates on or after 2025-04-01**), environment variables are available via the global `process.env`."
> "The `process.env` will be populated lazily the first time that `process` is accessed in the worker."
> "Note also that because secrets are a form of environment variable within the runtime, **secrets are also exposed via `process.env`.**"

- Your config: `nodejs_compat` is ON and `compatibility_date = 2025-09-24` (which is ≥ 2025-04-01). Therefore `nodejs_compat_populate_process_env` is **on by default** — you do NOT need to add it explicitly.
- Both **plaintext `[vars]`** and **Secrets** (`wrangler secret put`) are exposed via `process.env`. So `process.env.RESEND_API_KEY` resolves to your secret's value (not `undefined`).
- Value typing: text vars are exposed as strings; JSON vars that evaluate to a string are exposed parsed; JSON vars that are non-string are exposed as the raw JSON string.

### Caveats / best practice
- The **`env` argument** to `fetch(request, env, ctx)` is the canonical, portable, always-available mechanism and is unaffected by compat flags. Your `src/server.ts` already threads `env` through to the TanStack server entry — prefer reading the secret from `env` in request-scoped code where you have it.
- `process.env` is convenient for top-level/module init or library code that can't receive `env`. It is populated lazily on first `process` access.
- If you ever lower the compat date below 2025-04-01 or remove `nodejs_compat`, `process.env` population stops — then `process.env.RESEND_API_KEY` becomes `undefined`. You could re-enable explicitly by adding `"nodejs_compat_populate_process_env"` to `compatibility_flags`.
- Local dev: put the secret in `.dev.vars` (or `.env`) — both already covered by your `.gitignore` patterns per the docs' guidance.

## Q3 — Cloudflare Web Analytics: exact CSP domains, edge auto-injection, and strict-CSP coexistence

**Sources (Tier 1):**
- https://developers.cloudflare.com/web-analytics/faq/ (last updated May 20, 2026) — CSP section
- https://developers.cloudflare.com/web-analytics/get-started/ (last updated Apr 17, 2026) — automatic injection

### Exact CSP entries
- **`script-src`** must allow the beacon script:
  `script-src ... https://static.cloudflareinsights.com/beacon.min.js`
  (If you use the query-string form, include it: `.../beacon.min.js?token=$SITE_TOKEN`.)
- **`connect-src`** depends on setup mode:
  - **Automatic injection** (proxied site, Cloudflare injects at edge): beacon posts to **your own domain** `/cdn-cgi/rum`, so **`connect-src 'self'`** is sufficient. (No need for `cloudflareinsights.com`.)
  - **Manual embed** (you add the `<script>` yourself): beacon posts to **`cloudflareinsights.com`**, so add `connect-src ... cloudflareinsights.com`.

> Your prior belief ("connect-src needs https://cloudflareinsights.com") is only correct for **manual** embedding. For automatic injection it's `'self'`.

### Does automatic mode inject at the edge, and does strict CSP block it?
- **Yes**, for sites proxied through Cloudflare, automatic setup injects the beacon `<script>` at the edge into proxied HTML ("enabled by default"). You can switch a site to "Enable with JS Snippet installation" (manual) or disable in Manage Site.
- **Strict CSP (no `unsafe-inline`) is compatible.** The injected tag is an **external** script (`src="https://static.cloudflareinsights.com/beacon.min.js"`), NOT inline JS — so you only need `static.cloudflareinsights.com` in `script-src`; you do NOT need `'unsafe-inline'` for the beacon. Verbatim from FAQ:
  > "If you're using the automated injection … Cloudflare automatically includes an `integrity` attribute in the `<script>`." (SRI is added for you in automatic mode.)
- **Gotcha — injection silently fails if you over-cache HTML:**
  > "If you have a `Cache-Control` header set to `public, no-transform`, Cloudflare proxy will not be able to modify the original payload … the Beacon script will not be automatically injected."
  Don't send `no-transform` on your HTML document if you want edge injection.

### Recommended way to make strict CSP coexist with Web Analytics (this repo)
Two clean options:
1. **Automatic injection** (simplest): keep beacon out of your code; in your Worker CSP add `script-src ... https://static.cloudflareinsights.com` and keep `connect-src 'self'`. Ensure HTML isn't `no-transform`. Validate with `Content-Security-Policy-Report-Only` first (FAQ explicitly recommends this).
2. **Manual embed** (most deterministic for a Worker/SSR app): add the `<script defer src="https://static.cloudflareinsights.com/beacon.min.js" data-cf-beacon='{"token":"..."}'>` to your document, and set CSP `script-src ... https://static.cloudflareinsights.com` + `connect-src ... cloudflareinsights.com`. Note: manual embed can't be SRI-pinned (Cloudflare doesn't version-pin the beacon).

> Note: Your current CSP uses `script-src 'self' 'unsafe-inline'`. Whichever Web Analytics mode you pick, you must add `https://static.cloudflareinsights.com`. Long-term, dropping `'unsafe-inline'` (nonce/hash-based) is the real hardening win; Web Analytics does not require `'unsafe-inline'`.

## Q4 — Native Workers Rate Limiting binding on the FREE plan + current syntax; and WAF Rate Limiting Free-plan facts

### 4a. Workers native Rate Limiting binding
**Source (Tier 1):** https://developers.cloudflare.com/workers/runtime-apis/bindings/rate-limit/ (last updated Apr 23, 2026)

- The binding is documented as a stable Runtime API (the page no longer uses `unsafe.bindings` / `unstable_rate_limit`). **Current config uses the top-level `"ratelimits"` array.** Requires **Wrangler ≥ 4.36.0**.
- The docs place **no plan restriction** on the binding — it is usable on the **Workers Free plan** (it shares the same infra as WAF rate limiting but is configured per-Worker). There is no paid gate called out.
- **Locality caveat (important):** limits are **per Cloudflare data center**, not global, and the counter is "permissive, eventually consistent … not … an accurate accounting system." Fine for abuse mitigation on a contact form; not for hard quotas.

**Current `wrangler.jsonc` syntax:**
```jsonc
{
  "main": "src/server.ts",
  "ratelimits": [
    {
      "name": "CONTACT_RATE_LIMITER",
      // unique positive integer per account, specified as a STRING
      "namespace_id": "1001",
      "simple": {
        "limit": 5,        // tokens allowed per period, per CF location
        "period": 60       // seconds — MUST be 10 or 60 (no other values)
      }
    }
  ]
}
```

**API usage (`.limit({ key })` returns `{ success }`):**
```ts
interface Env { CONTACT_RATE_LIMITER: RateLimit }

const { success } = await env.CONTACT_RATE_LIMITER.limit({ key: someStableKey });
if (!success) return new Response("429 Too Many Requests", { status: 429 });
```
- `key` is any string you choose. Cloudflare recommends a stable identifier (API key, user/tenant ID, route). They explicitly **advise against keying on IP** (`cf-connecting-ip`) because many users share IPs — though for an anonymous public contact form, IP or IP+route is often the only signal available; just understand the false-positive risk.
- Two bindings with the same `namespace_id` (even across Workers) share counters; use a unique `namespace_id` to isolate.
- Not visible in the dashboard; observe 429s via Workers Logs/Traces or an Analytics Engine binding.

### 4b. WAF Rate Limiting Rules — Free plan
**Source (Tier 1):** https://developers.cloudflare.com/waf/rate-limiting-rules/ (last updated Apr 22, 2026), "Availability" table.

**Your statement is confirmed accurate.** Free plan:
- **Number of rules: 1**
- **Available fields in rule expression: Path, Verified Bot** (i.e., effectively path-only matching for your purposes — no Host/URI/Query/Method/IP/User-Agent matching on Free)
- **Counting characteristics: IP** (counts requests by client IP)
- **Counting period: 10 s only** (fixed)
- **Mitigation timeout: 10 s only** (fixed)
- No cache exclusion, no custom counting expression on Free.

(Pro raises to 2 rules and adds Host/URI/Full URI/Query fields; Business 5 rules; Enterprise 100+.)

**Takeaway for this repo:** The native Workers binding (4a) is the better tool for app-aware contact-form throttling on Free, because the single WAF Free rule is coarse (path + IP, 10s window). Use the WAF rule as a blunt outer guard if desired.

## Q5 — Turnstile Free plan limits + current siteverify endpoint

**Sources (Tier 1):**
- https://developers.cloudflare.com/turnstile/plans/ (last updated Apr 16, 2026)
- https://developers.cloudflare.com/turnstile/get-started/server-side-validation/ (last updated May 5, 2026)

### Free plan limits (verbatim from plans table)
- **Number of widgets: Up to 20 widgets** (per account).
- **Unlimited challenges (traffic / verification requests): Yes** — there is **no monthly verification cap** on Free.
- **All widget types: Yes.**
- **Hostname management: 10 hostnames per widget.**
- **Analytics lookback: 7 days maximum.**
- **Pre-clearance support: Yes.**
- **WCAG 2.2 AAA compliance: Yes.**
- **Ephemeral IDs: No** (Enterprise only).
- **Off-label / remove Cloudflare branding: No** (Enterprise only).

### Siteverify
- **Endpoint:** `POST https://challenges.cloudflare.com/turnstile/v0/siteverify`
- Accepts `application/x-www-form-urlencoded` or `application/json`; always returns JSON.
- Required params: `secret` (server secret key), `response` (client token). Optional: `remoteip`, `idempotency_key` (UUID for safe retries).
- **Token rules:** valid **300 s (5 min)**, **single-use** (replays return `timeout-or-duplicate`), max length **2048 chars**.
- Server-side validation is **mandatory** — the client widget alone is not protection. Call siteverify **only** from the backend (never expose the secret key client-side).
- Success response includes `success`, `challenge_ts`, `hostname`, `action`, `cdata`. You can additionally assert `action` and `hostname` match expected values.

**Fit for this repo:** Turnstile Free is more than sufficient for one contact form (1 widget of your 20; unlimited verifications). In your Worker contact handler, read `cf-turnstile-response` from the form, POST to siteverify with your `TURNSTILE_SECRET` (a Worker Secret — resolvable via `env` or `process.env` per Q2), and reject on `!success`.

## Q6 — HSTS: dashboard vs response-header-in-code, conflicts, and preload requirements

**Sources (Tier 1):**
- https://developers.cloudflare.com/ssl/edge-certificates/additional-options/http-strict-transport-security/ (last updated Apr 17, 2026)
- https://hstspreload.org/ (primary source for preload-list submission requirements)

### Dashboard vs code
- HSTS is available on **all plans including Free**. Dashboard path: **SSL/TLS -> Edge Certificates -> HTTP Strict Transport Security (HSTS) -> Enable HSTS**. Settings: `max-age` (Disable, or 1–12 months), `includeSubDomains`, `Preload`, `No-Sniff`.
- The docs explicitly allow setting it yourself instead:
  > "you can add the appropriate HSTS header at the origin, or use a response header transform rule."
  So a Worker `Strict-Transport-Security` response header is a fully supported alternative.

**Recommendation: pick ONE source of truth.**
- **Dashboard (recommended for zone-wide HSTS):** centrally managed; Cloudflare serves it on all HTTPS responses and **omits it on plain-HTTP requests automatically** (correct HSTS behavior). Best when you want the whole zone covered regardless of which Worker/route answers.
- **Worker code:** good if you want HSTS coupled to your app/deploy and versioned in git. If you go this route, **only emit it on HTTPS** (don't send HSTS over HTTP) and keep it consistent across routes.

### Conflict if both are set?
- Setting HSTS in **both** the dashboard and your Worker can produce a **duplicate `Strict-Transport-Security` header**, or one layer overriding the other at the edge. Browsers will generally honor a present valid HSTS policy, but duplicated/conflicting `max-age` values are a config smell and make audits (and preload submission checks) ambiguous. **Avoid configuring it in two places.** Choose dashboard *or* code.
- Practical note for this repo: if you enable dashboard HSTS, **remove** `Strict-Transport-Security` from both `_headers` and any Worker code to avoid duplication. If you instead set it in the Worker (`applyHeaders`), leave dashboard HSTS off.

### hstspreload.org submission requirements (verbatim from the source)
To be eligible for the preload list, serve an HSTS header on the base domain for HTTPS requests with:
1. **`max-age` must be at least `31536000` seconds (1 year).**
2. **`includeSubDomains` directive must be specified.**
3. **`preload` directive must be specified.**
4. If you serve an additional redirect from your HTTPS site, that redirect must still carry the HSTS header.

Plus the surrounding requirements:
- Serve a valid certificate.
- **Redirect HTTP -> HTTPS on the same host** (if listening on port 80).
- **Serve all subdomains over HTTPS** (in particular `www` if a DNS record exists), each also satisfying HSTS.

Cloudflare's own note aligns (and is slightly stricter on UI granularity):
> "A minimum Max Age Header of **12 months** is required for inclusion in HSTS preload lists." (Chrome, Firefox/Mozilla, and Safari all use the Chrome preload list.)

**Your current header value** (`max-age=31536000; includeSubDomains; preload`) **meets the preload requirements** — BUT remember (Q1) it currently only ships on static assets, not the SSR HTML, so before submitting to hstspreload.org you must ensure HSTS is present on the document response (via dashboard or Worker code). **Preload is a near-permanent commitment** — every current and future subdomain must stay HTTPS-only for the `max-age` duration; only opt in once you're certain.
