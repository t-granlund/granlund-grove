##  Headline finding (a real bug in this repo)

Your security headers strategy is **half-broken right now**, and Cloudflare's own docs confirm why.

`public/_headers` sets `Content-Security-Policy`, `Strict-Transport-Security`, and `Permissions-Policy`. But per the official Workers Static Assets **Headers** docs:

> **"Custom headers defined in the `_headers` file are not applied to responses generated by your Worker code, even if the request URL matches a rule. If you use a server-side rendered (SSR) framework … you will likely need to attach any custom headers you wish to apply directly within that Worker script."**

Your SSR HTML document (`/`, `/api/contact` errors, every route) is generated **by the Worker**, so **`_headers` does NOT apply to it.** Your `src/server.ts` `applyHeaders()` only adds `X-Content-Type-Options`, `X-Frame-Options`, and `Referrer-Policy` — it does **NOT** set CSP, HSTS, or Permissions-Policy.

**Net effect: your SSR HTML pages currently ship with NO Content-Security-Policy, NO Strict-Transport-Security, and NO Permissions-Policy.** Those three only land on *static asset* responses (fonts, favicon, PDFs, og image). The code comment in `server.ts` ("_headers file also sets these") is misleading for exactly these three headers.

**Fix:** move CSP / HSTS / Permissions-Policy into `applyHeaders()` in `src/server.ts`. See `recommendations.md`.

## Quick answers

| # | Question | Verdict |
|---|----------|---------|
| 1 | Does `_headers` apply to SSR/Worker responses? | **NO** — static assets only. `_headers`/`_redirects` ARE supported for Workers Static Assets, but never apply to Worker-generated responses. Set CSP/HSTS in Worker code. |
| 2 | Will `process.env.RESEND_API_KEY` resolve? | **YES.** `nodejs_compat` + compat date ≥ 2025-04-01 enables `nodejs_compat_populate_process_env` by default. Secrets are exposed via `process.env` too. (`env` arg still works and is the portable default.) |
| 3 | Web Analytics CSP domains | `script-src https://static.cloudflareinsights.com` (+`/beacon.min.js`); **automatic** mode → `connect-src 'self'`; **manual** mode → `connect-src cloudflareinsights.com`. Strict CSP (no `unsafe-inline`) is fine: injected script is external + carries an `integrity` (SRI) attr. |
| 4 | Native Rate Limit binding on Free? | Binding is GA, uses the stable top-level `"ratelimits"` key (NOT `unsafe.bindings`/`unstable_*`), needs Wrangler ≥ 4.36.0, no plan gate in docs → usable on Free. WAF Rate Limiting on Free = **1 rule, Path + Verified Bot only, count by IP, 10s period** — confirmed. |
| 5 | Turnstile Free limits | **20 widgets/account, UNLIMITED verifications**, 10 hostnames/widget, 7-day analytics. Siteverify: `POST https://challenges.cloudflare.com/turnstile/v0/siteverify`. Tokens: 300s, single-use, ≤2048 chars. |
| 6 | HSTS dashboard vs code | Prefer **one source of truth.** Dashboard (Edge Certificates) is zone-wide & safest (auto-stripped on HTTP). Don't set in both. Preload needs `max-age ≥ 31536000` + `includeSubDomains` + `preload` + HTTP→HTTPS redirect + all subdomains on HTTPS. |
