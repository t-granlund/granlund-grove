# Raw verbatim excerpts (for audit)

## Q1 — Workers Static Assets / Headers
Source: https://developers.cloudflare.com/workers/static-assets/headers/

> "The default response headers served on static asset responses can be overridden, removed, or added to, by creating a plain text file called `_headers` without a file extension, in the static asset directory of your project. This file will not itself be served as a static asset, but will instead be parsed by Workers and its rules will be applied to static asset responses."

> "Custom headers defined in the `_headers` file are not applied to responses generated by your Worker code, even if the request URL matches a rule defined in `_headers`. If you use a server-side rendered (SSR) framework, have configured `assets.run_worker_first`, or otherwise use a Worker script, you will likely need to attach any custom headers you wish to apply directly within that Worker script."

> "If you are server-side rendering (SSR) or using a Worker to generate responses in any other way and wish to attach security headers, the headers should be sent from the Worker's `Response` instead of using a `_headers` file."

## Q2 — Environment variables / nodejs_compat
Source: https://developers.cloudflare.com/workers/configuration/environment-variables/

> "When you enable `nodejs_compat` and the `nodejs_compat_populate_process_env` compatibility flag (enabled by default for compatibility dates on or after 2025-04-01), environment variables are available via the global `process.env`."

> "The `process.env` will be populated lazily the first time that `process` is accessed in the worker."

> "Note also that because secrets are a form of environment variable within the runtime, secrets are also exposed via `process.env`."

## Q3 — Web Analytics CSP
Source: https://developers.cloudflare.com/web-analytics/faq/

> "You'll first need to permit our script to execute by adding it to your script-src directive: `script-src [...existing values...] https://static.cloudflareinsights.com/beacon.min.js`"

> "For automatic injection, this will be the same domain, so ensure your connect-src includes 'self': `connect-src [...existing values...] 'self'`"

> "For manual embedding, this script instead connects to cloudflareinsights.com, so ensure that's included instead: `connect-src [...existing values...] cloudflareinsights.com`"

> "If you're using the automated injection (i.e. not the manually-embedded script approach mentioned above), Cloudflare automatically includes an integrity attribute in the <script>."

> "If you have a Cache-Control header set to public, no-transform, Cloudflare proxy will not be able to modify the original payload ... the Beacon script will not be automatically injected."

Source: https://developers.cloudflare.com/web-analytics/get-started/
> "Your website is now using Web Analytics through the automatic setup, which is enabled by default." (for sites proxied through Cloudflare)

## Q4a — Workers Rate Limiting binding
Source: https://developers.cloudflare.com/workers/runtime-apis/bindings/rate-limit/

> "You must use version 4.36.0 or later of the Wrangler CLI."

Config (verbatim):
```jsonc
"ratelimits": [
  {
    "name": "MY_RATE_LIMITER",
    "namespace_id": "1001",
    "simple": { "limit": 100, "period": 60 }
  }
]
```
> "Period: the duration of the period, in seconds. Must be either 10 or 60"

API (verbatim):
```js
const { success } = await env.MY_RATE_LIMITER.limit({ key: pathname })
```
> "Rate limits that you define and enforce in your Worker are local to the Cloudflare location that your Worker runs in."

## Q6 — HSTS
Source: https://developers.cloudflare.com/ssl/edge-certificates/additional-options/http-strict-transport-security/
> "To enable HSTS on a specific subdomain only ... you can add the appropriate HSTS header at the origin, or use a response header transform rule."
> "A minimum Max Age Header of 12 months is required for inclusion in HSTS preload lists." (Chrome, Firefox/Mozilla, and Safari use the Chrome preload list.)
- Max Age Header range in dashboard: "Disable, or a range from 1 to 12 months".

Source: https://hstspreload.org/ (Submission Requirements)
> "Serve an HSTS header on the base domain for HTTPS requests:"
> "The max-age must be at least 31536000 seconds (1 year)."
> "The includeSubDomains directive must be specified."
> "The preload directive must be specified."
Plus: "Serve a valid certificate." / "Redirect from HTTP to HTTPS on the same host, if you are listening on port 80." / "Serve all subdomains over HTTPS ... In particular, you must support HTTPS for the www subdomain if a DNS record for that subdomain exists."
